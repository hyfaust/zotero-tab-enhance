# Zotero 垂直标签栏分阶段执行清单

## Summary
- 路线固定为“插件侧维护 tab 模型 + 左侧垂直 sidebar UI + 与原生 `Zotero_Tabs` 双向同步”，不再继续直接改造原生水平标签 DOM 作为主方案。
- 默认目标：
  - 状态持久化恢复。
  - 虚拟组优先。
  - 尽量使用 `zotero-plugin-template` 生命周期与 `zotero-plugin-toolkit` 的 XUL/UI 能力。
- 每一阶段都要求“可单独验收、可停下使用、不会破坏现有右键增强功能”。

## Current Progress
- Phase 0：已完成。
- Phase 1：已完成，`_tabs` 作为运行时 tab 身份源，`getState()` 只补充标题、itemID、选中态等状态。
- Phase 2：已完成基础侧栏壳层，支持显示、收起、伸缩与高亮当前标签。
- Phase 3：已完成 v1，已支持点击切换、关闭、搜索、右键菜单；`Ctrl/Cmd` / `Shift` 多选仍待补。
- Phase 4：已完成 v1，已支持侧栏单标签拖拽排序并联动 `Zotero_Tabs.move(...)`；多选拖拽与更多边界优化仍待补。
- Phase 5：已完成 v1，已支持虚拟分组创建、加入、移除、重命名、折叠、改色与解散；当前分组语义为“虚拟副本组”，不会把真实标签从主列表移走；持久化仍待后续阶段补全。
- Phase 6：已完成 v1，已支持虚拟成员点击恢复、打开组内全部、关闭其他并打开本组、仅展开本组；批量命令的更细致恢复策略仍可继续优化。
- Phase 7：已完成 v1，已支持默认/最近/类型三种展示视图切换；当前聚合视图只影响展示，不参与拖拽排序。
## Phase Checklist

### Phase 0. 预研与接口基线
- 目的：把“哪些接口可依赖、哪些只能适配”先锁死，避免后面返工。
- 产出：
  - 确认 Zotero 原生 tab 事实来源：`Zotero_Tabs` 的 tab 列表、选中、关闭、选择、排序入口。
  - 确认插件内稳定模型字段和 key 方案。
  - 记录需要参考 Zotero 源码的位置，尤其是 tab move / update / reader open 逻辑。
  - 记录 `dev` 分支哪些实验结论可复用，哪些明确废弃。
- 代码层面计划：
  - 新建设计文档注释或模块说明，不动现有行为。
  - 给现有 `tabEnhance` 保留接口，避免后续迁移时断层。
- 验收标准：
  - 能明确列出“追踪、选择、关闭、排序、打开虚拟成员”分别调用哪层 API。
  - 明确哪些调用是公开 API，哪些是内部 API 包装后使用。

### Phase 1. 内部模型与追踪服务
- 当前状态：已完成。
- 目的：先让插件拥有稳定的 tab 真相副本。
- 产出：
  - `TabTrackerService` 或等价模块。
  - `TrackedTab`、`SidebarState`、基础 store 类型。
  - 初始化全量扫描 + reconcile 机制。
- 实现内容：
  - 在主窗口加载时读取当前全部 tabs，生成内部列表。
  - 监听 tab open / close / select / reorder 相关变化并增量同步。
  - 给每个 tab 统一生成展示字段：
    - 标题
    - 类型
    - itemID / parentItemID
    - 原生位置
    - 最近激活时间
    - 是否真实打开
  - library tab 仍由 Zotero 原生保留，但不进入侧边栏展示列表。
- tab 身份源固定使用 `Zotero_Tabs._tabs` 的运行时 `id`；`getState()` 只作为 `itemID`、标题、选中态等补充状态来源。
- 已验证：侧边栏点击可以切换到对应原生 tab。
- 验收标准：
  - 打开/关闭/切换标签后，内部模型始终与原生 tabs 一致。
  - 多窗口下每个窗口 tracker 独立。
  - 当前插件已有右键功能不回归。

### Phase 2. Sidebar 壳层与基础渲染
- 当前状态：已完成基础版本。
- 目的：先把左侧垂直栏“立起来”，哪怕功能还不全。
- 产出：
  - `VerticalTabSidebar` 或等价模块。
  - 可注入/可移除的 XUL 容器。
  - 展开/收起状态与宽度状态。
- 实现内容：
  - 在 Zotero 主窗口左侧插入 sidebar 容器，位置固定且不覆盖原生核心区域。
  - 使用 toolkit/XUL 创建：
    - 外层容器
    - 顶部工具条
    - 收起/展开按钮
    - 列表滚动区
  - 展开态显示标题与次信息；收起态显示图标和选中状态。
  - 当前选中 tab 在侧栏高亮。
- 已实现基础点击交互闭环，但关闭按钮/搜索/右键菜单仍留在 Phase 3。
- 验收标准：
  - 侧栏可稳定显示、隐藏、伸缩。
  - 打开多个标签时侧栏能正确列出并高亮当前标签。
  - 窗口关闭/插件卸载时 sidebar 能干净移除。

### Phase 3. 基础交互闭环
- 当前状态：已完成 v1，剩余多选能力待补。
- 目的：让侧栏具备最小可用性，能替代一部分水平标签切换。
- 产出：
  - 点击切换、关闭、搜索、右键菜单。
  - 现有 `showInFilesystem` / `reloadTab` / `copyReference` 融入侧栏操作。
- 实现内容：
  - 点击侧栏项切换到原生 tab。
  - 关闭按钮关闭真实 tab。
  - 搜索框按标题/元信息过滤列表。
  - 多选基础能力：支持 `Ctrl/Cmd` 与 `Shift` 范围选择。
  - 右键菜单统一走命令控制器，不在 UI 层直接写业务。
- 验收标准：
  - 用户可以只通过侧栏完成“找 tab、切 tab、关 tab、搜索 tab、执行右键动作”。
  - 搜索过滤不会打乱真实顺序。
  - 多选状态与单选状态切换逻辑稳定。

### Phase 4. 原生排序联动
- 当前状态：已完成 v1，已接入单标签拖拽排序与原生顺序联动。
- 目的：解决最关键的“拖动排序必须作用到原始水平标签排序”。
- 产出：
  - `TabCommandController.moveOpenTabs(...)`。
  - Sidebar 拖拽排序。
  - 原生顺序回写与冲突处理。
- 实现内容：
  - 侧栏拖拽只允许对真实打开 tabs 生效。
  - 拖拽结束后调用封装过的原生 tab move 逻辑。
  - 拖拽过程中展示插入提示，不直接修改原生 DOM。
  - 处理非法目标：
    - library tab 不可移动
    - 跨过滤视图拖拽时按真实顺序解释
  - reorder 后触发 tracker reconcile，保证最终一致。
- 验收标准：
  - 从侧栏拖动后，水平标签顺序同步变化。
  - 从水平标签拖动后，侧栏顺序也能同步回来。
  - 非法拖拽不会导致原生 tab 列表损坏。

### Phase 5. 虚拟分组 v1
- 当前状态：已完成 v1，组内拖拽与持久化留到后续阶段。
- 目的：在不强依赖真实打开标签的前提下实现最核心的产品差异化。
- 产出：
  - `TabGroupStore`
  - `VirtualGroup`
  - 组创建/删除/重命名/折叠/改色
- 实现内容：
  - 基于当前选中的真实 tab 或虚拟条目创建组。
  - 组成员保存为稳定 key，不保存脆弱的 DOM 位置。
  - 分组成员是侧栏中的虚拟副本，不会把真实打开标签从默认主列表中抽离。
  - 组内显示计数、颜色条、折叠状态。
  - 支持：
    - 新建组
    - 加入现有组
    - 从组移除
    - 解散组
  - 组头和组成员完全在侧栏模型层实现，不再注入原生水平标签条。
- 验收标准：
  - 组折叠/展开稳定。
  - 加入组后，真实标签仍保留在默认列表中，组内只新增虚拟副本。
  - 组操作不会直接破坏原生 tab 顺序。
  - 已关闭的成员仍可作为虚拟成员留在组里。

### Phase 6. 虚拟成员激活与组命令
- 当前状态：已完成 v1，批量命令容错与更细致的恢复提示可继续优化。
- 目的：让“虚拟组优先”真正可用。
- 产出：
  - 组级命令与成员激活逻辑。
- 实现内容：
  - 点击虚拟成员时：
    - 若真实 tab 已存在，则切换过去。
    - 若不存在，则按 item/attachment 信息打开对应 reader tab。
  - 组级菜单支持：
    - 打开组内全部
    - 关闭其他并打开本组
    - 仅展开本组
  - 为避免性能风险，批量打开采用串行或受控节流。
- 验收标准：
  - 虚拟成员可可靠恢复成真实 tab。
  - “关闭其他并打开本组”执行后，侧栏与原生 tabs 状态一致。
  - 批量操作失败时不会留下半损坏状态。

### Phase 7. 聚合视图与展示增强
- 当前状态：已完成 v1，已支持默认/最近/类型视图切换。
- 目的：把“分组”之外的浏览效率再往前推一步，但仍控制复杂度。
- 产出：
  - 可切换聚合视图 v1。
- 实现内容：
  - 至少提供一种轻量聚合：
    - 按最近打开
    - 或按条目来源
    - 或按类型
  - 聚合视图只影响展示，不改变真实 tab 顺序。
  - 收起态显示组/聚合计数。
  - 次信息展示规则固定一版，如“作者/期刊/年份”或“父条目标题”二选一。
- 验收标准：
  - 聚合视图与默认列表视图切换稳定。
  - 聚合不影响拖拽排序事实源。
  - 列表渲染在较多标签下仍可接受。

### Phase 8. 持久化、恢复与偏好设置
- 目的：让功能从“能用”变成“日常可用”。
- 产出：
  - 偏好设置项
  - 存储与恢复逻辑
- 实现内容：
  - 持久化：
    - sidebar 显隐
    - 宽度
    - 搜索/视图偏好
    - groups 数据
    - 组折叠状态
    - 最近激活时间或排序辅助信息
  - 插件启动时恢复 sidebar 和 group 状态。
  - 若某些组成员对应条目已不存在，恢复时自动跳过并清理。
- 验收标准：
  - 重启 Zotero 后用户布局和分组保持一致。
  - 数据损坏时可安全回退到空状态，不阻塞插件加载。
  - 偏好设置能控制功能开关。

### Phase 9. 回归、兼容与发布准备
- 目的：把实验功能收敛为可维护版本。
- 产出：
  - 自动化测试与手工回归清单。
  - 文档更新与风险说明。
- 实现内容：
  - 为 tracker/store/command 层补单元测试。
  - 补启动测试和关键状态恢复测试。
  - 回归多窗口、reader tab、关闭组、异常数据恢复。
  - 更新 README，说明“虚拟组”语义与原生标签的关系。
- 验收标准：
  - 基础测试通过。
  - 主要流程无明显 UI 锁死、排序错乱、状态不同步问题。
  - 用户文档能解释该功能与浏览器 tab group 的差异。

## Milestones
- M1 可演示版：完成 `Phase 1-3`
  - 已能追踪 tab、显示侧栏、点击切换、搜索与基本操作。
- M2 可替代日常切换版：完成 `Phase 4-5`
  - 已能拖拽联动原生排序，并具备虚拟分组。
- M3 可长期使用版：完成 `Phase 6-9`
  - 已有恢复、组命令、聚合视图、测试与文档。

## Assumptions
- 当前实现基线以 `sidebar`/主线模板结构为准，`dev` 分支只作为实验参考，不直接并入其 DOM 分组方案。
- 第一版不做复杂 PDF 预览浮层，不做缩略图渲染。
- `getState()` 不再被视为稳定 tab 身份来源；运行时 tab 身份一律以 `_tabs` 为准。
- 如果 Zotero 内部排序 API 不稳定，则优先封装兼容层，而不是把拖拽功能提前扩展到虚拟成员。
- 若阶段中发现 Zotero 内部接口限制明显，优先保证“追踪正确、恢复稳定”，次优先级才是复杂聚合与高级展示。
